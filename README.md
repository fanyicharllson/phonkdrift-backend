# PhonkDrift Backend (Enterprise Microservices Engine)

PhonkDrift is a high-performance, low-latency social music streaming platform engineered specifically for the underground Phonk community. The backend is architected using a cloud-native, event-driven polyrepo approach built entirely in **Go (Golang)**, leveraging high-speed synchronous **gRPC** protocols for internal communications and asynchronous message queuing via **RabbitMQ**.

---

## 🏛️ Repository Architecture & Folder Use Case

The directory layout adheres strictly to the **Standard Go Project Layout**. This separation ensures that infrastructural code, runtime configurations, and core business rules are decoupled.

* `api/proto/` - **The Source of Truth Contract Layer.** Contains the raw `.proto` definition files (`auth.proto`, `track.proto`). These files define the exact structure of request/response payloads and gRPC server capabilities. They are shared directly with multi-client teams (Flutter, Web, Desktop) to compile uniform client-side network packages.
* `cmd/` - **Application Bootstrapping / Entry Points.** Contains the `main.go` file for each independent microservice daemon (`gateway`, `auth-service`, `track-service`). Code here is minimal; its sole responsibility is to ingest environment configurations via `.env` files, establish database pools, and ignite network listeners.
* `internal/` - **Protected Domain Logic Layer.** This directory houses your actual intellectual property. Code inside `internal/` is structurally guarded by Go compilation rules and cannot be imported by external applications.
    * `internal/config/` - Houses the configuration loading engine powered by Viper to parse variables into memory types safely.
    * `internal/[service-name]/delivery/` - **The Transport Layer.** Implements the generated gRPC server interfaces. It unpacks incoming network bytes and routes commands downward.
    * `internal/[service-name]/usecase/` - **The Pure Business Logic Layer.** Contains structural workflows, validation conditions, and calculations. It is completely independent of network delivery schemas or external database providers.
    * `internal/[service-name]/repository/` - **The Data Abstraction Layer.** Handles raw input/output operations with external infrastructure providers (Supabase PostgreSQL, Upstash Redis, or system utilities like `yt-dlp`).
* `pb/` - **Generated Machine Code Binaries.** Houses the auto-generated Go files generated by running `make proto`. **Never edit files in this folder manually.**
* `docker-compose.yml` - Orchestrates the containerized dependencies locally for development or directly onto a low-cost production DigitalOcean virtual machine.
* `Makefile` - Task automation interface to eliminate typing long shell lines during code compilation cycles.

## 🔄 End-to-End Request & Data Flow Architecture

When a mobile user interacts with the Flutter application, data moves seamlessly across synchronous and asynchronous layers:



### Phase 1: Client to API Gateway (Secure Entry)
1. The Flutter mobile client instantiates a persistent HTTP/2 connection and sends a binary-encoded gRPC message payload (e.g., `GetStreamURLRequest`) directly to the public-facing port of the **Go API Gateway**.
2. The Gateway acts as a reverse proxy, intercepting the stream to perform edge checks (rate-limiting, SSL termination, and preliminary API authentication validation tokens).

### Phase 2: Internal Synchronous Coordination (High-Speed gRPC Highway)
3. The Gateway dynamically translates the public client call into an internal gRPC request and establishes a fast, point-to-point network connection to the targeted back-end container—the **Track Service**.
4. To fulfill the search or track request, the **Track Service** executes a synchronous query to its persistent database array. If the query requires user authorization details, the Track Service triggers a secondary synchronous gRPC request to the **Auth Service** to assert identity profiles.

### Phase 3: The Circuit-Breaker Worker & Stream Resolve
5. The Track Service business logic calls upon the embedded `yt-dlp` streaming abstraction layer to fetch direct CDN streams. This entire cycle is safely wrapped inside a **Circuit Breaker state engine**. If downstream network timeouts spike, the fuse trips immediately to bypass the failure and serve cached local alternatives, keeping the client runtime completely stable.
6. The finalized stream payload matches the Proto interface and cascades back up the gRPC chain through the API Gateway, returning an instant, highly optimized streaming link string directly to the mobile device.

### Phase 4: Asynchronous Cascades (Non-blocking Operations)
7. Simultaneously, if the system flags the track execution as a milestone event, the Track Service fires a non-blocking background notification message into the **RabbitMQ Event Bus** (`track.trending`).
8. The main streaming process is completed immediately without waiting. Meanwhile, the independent **Notification Broker Service** consumes the event off the queue asynchronously and handles mass push notifications via Firebase safely in the background.
